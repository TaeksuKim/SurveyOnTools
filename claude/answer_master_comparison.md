# master-comparison.md 사용자 질문에 대한 답변 (Claude)

작성자: Claude
기준일: 2026-06-17
범례: ● 1급(native·성숙) · ◐ 부분 지원(제한적·간접) · ○ 미지원/미확인

> **검증 caveat (AGENTS.md §6):** 아래 답변 중 *개념 설명*(권한·샌드박싱·rewind·라우팅 등)은
> 제가 자신 있게 기술할 수 있는 영역입니다. 반면 특정 제품의 *내부 구현 주장*(예: Cline의
> 메모리 저장 방식, Hermes의 SQLite 계층 등)은 공개 문서가 제한적이라 일부 추정이 섞일 수
> 있습니다. 마스터 표의 일부 등급이 **공식 문서(마케팅) 기준으로 과대평가**되어 있다는
> 사용자님의 지적에 전반적으로 동의하며, 그 부분은 명시했습니다.

---

## 들어가며 — 사용자님 지적의 핵심

질문 전체를 관통하는 문제의식은 **"문서 스펙(claim) ≠ 실사용 체감(reality)"** 입니다.
이는 정확한 지적이고, 마스터 표 작성 과정의 구조적 약점을 정확히 찌릅니다. 취합 과정에서
특수목적 3종과 Cline은 1차 출처 접근이 어려워 **공식 문서의 자기 선전을 액면가로 ● 처리한
셀**이 존재합니다. 아래에서 해당 셀들을 솔직하게 재평가하고, v0.6 하향안을 제시합니다.

Antigravity·Gemini 두 답변과 **결론은 대체로 일치**합니다(Cline 메모리/압축/개인화 하향).
저는 거기에 ① Claude Code 자체 기능에 대한 정확한 기술 설명과 ② Cline 멀티에이전트 건은
오히려 **과소평가**일 수 있다는 반대 방향 보정을 추가합니다.

## 1.1.A Core

### Q1. Cline의 Agentic Memory가 정말 강점(●)인가? 근거는?
**A. 사용자님 체감이 맞습니다. ● → ◐ 하향을 권고합니다.**

Cline의 메모리는 소위 **Memory Bank** 패턴 — `memory-bank/` 아래 마크다운 파일들
(projectbrief, activeContext, progress 등)을 **모델이 직접 읽고 다시 쓰도록 지시하는
프롬프트 규칙(.clinerules)** 입니다. 즉 프레임워크가 자동으로 임베딩·검색·write-back을
수행하는 구조가 아니라, **"문서를 읽고 갱신하라"는 규약에 모델의 성실성을 의존**합니다.

→ 한계: ① 컨텍스트가 길어지면 갱신 지시 자체가 압축에 밀려 누락되고, ② 자동 회상(recall)이
아니라 매 세션 수동/규약적 로딩이라 항목이 커질수록 신뢰도가 떨어집니다. **"설계는 영속이나
보장은 best-effort"** 라서 ●는 과합니다. Antigravity·Gemini도 동일 결론(◐/○)입니다.

비교 기준점: Claude Code도 `CLAUDE.md` 파일 메모리(역시 파일 주입형)지만 ●로 둔 이유는
계층(user/project/local)·`#` 즉석 저장·`/memory` 편집 등 **갱신 동선이 1급 시스템으로
정비**돼 있기 때문입니다. Cline은 그 정비 수준이 낮아 같은 "파일 메모리"라도 등급 차가 납니다.

### Q2. Cline의 Context Compaction이 실제로 잘 되는가?
**A. 사용자님의 pain point가 맞습니다. ● → ◐ 하향을 권고합니다.**

Cline의 `Auto Compact`는 컨텍스트 한도 근처에서 **대화를 요약해 치환**합니다. 문제는 Cline이
다양한 오픈/소형 컨텍스트 모델까지 지원하려다 보니 압축이 **모델·시점 의존적**이고,
요약 과정에서 ① 디버깅 로그·에러 메시지의 세부, ② 초반 제약/결정사항이 증발하는
"중간 망각(lost-in-the-middle)"이 자주 발생합니다. 게다가 압축 직후 다시 컨텍스트를 채우며
**토큰을 재낭비**하는 사이클이 체감 저하의 원인입니다.

→ "압축 기능의 존재"는 사실이나 "압축의 품질"은 ●가 아닙니다. 이건 표의 정의가
*기능 유무*에 치우쳐 *품질*을 못 담은 사례라, **◐ + "단순 요약 중심, 세부 유실 잦음" 주석**이
정직합니다. (Claude Code를 ●로 둔 근거는 다계층 압축 + 도구결과 정리 + 파일로의 오프로딩
조합이며, 이 역시 완벽하진 않으나 전략의 구조화 수준이 다릅니다.)

### Q3. Codex CLI는 왜 멀티/서브 에이전트가 약한가(◐)?
**A. "병렬 실행 ≠ 이질적 역할 팀"이기 때문입니다. ◐가 적정합니다.**

Codex는 병렬 스레드/동시 작업은 잘 하지만, 이는 대체로 **동일 에이전트의 다중 인스턴스가
도구를 병렬 호출**하는 형태입니다. 마스터 표의 ●(멀티에이전트) 기준은 **서로 다른 시스템
프롬프트·도구·권한을 가진 이질적 서브에이전트를 스폰하고 조율(orchestration)** 하는 것입니다
— Claude Code의 subagent/Agent Teams, Antigravity의 Mission Control(Manager/Writer/Critic/
Tester)이 그 예입니다. Codex는 *역할 분담형 팀 오케스트레이션*이 네이티브가 아니라 ◐입니다.
(Antigravity·Gemini와 동일한 판단입니다.)

### Q4. 멀티모델 라우팅이란? (자동 모델 선택?)
**A. 네, "작업 종류·난이도에 따라 프레임워크가 모델을 자동 전환"하는 것입니다.**

핵심은 **"고를 수 있다"(수동 선택)가 아니라 "알아서 고른다"(자동)** 입니다.
- 단순 파일 탐색/오타 수정 → 저렴·빠른 모델(예: Haiku, Flash)
- 복잡한 설계/리팩토링 → 고성능 모델(예: Opus, GPT-5.5)
를 **에이전트가 의도를 분류해 라우팅**합니다. oh-my-openagent의 IntentGate가 대표적이라
●입니다. 반면 Claude Code/Codex/Antigravity는 모델을 수동 전환하거나 소형 모델을 보조로
쓰는 정도라 **단일 벤더 내 부분 자동화 = ◐** 입니다. (이 정의는 표 §3에도 명시돼 있습니다.)

### Q5. Cline SDK는 멀티에이전트가 핵심인데 왜 점수(○)가 낮은가?
**A. 여기는 오히려 과소평가 가능성이 있습니다. ○ → ◐(SDK 한정) 상향을 제안합니다.**

이 항목은 사용자님이 *반대 방향*으로 지적하신 유일한 건이고, 일리가 있습니다. 평가가 ○인
이유는 비교 대상이 **"사용자가 설치해 쓰는 Cline 앱(VS Code 확장/CLI)"** 이고, 그 기본
워크플로는 `Plan`↔`Act`를 오가는 **강력한 단일 에이전트**라서, 백그라운드에서 다수 서브
에이전트가 자율 분업하는 기능이 **런타임 기본 탑재가 아니기** 때문입니다.

다만 `@cline/sdk`(5계층 아키텍처)가 멀티에이전트 *구축 기반*을 제공하는 것은 사실이므로,
다른 도구도 SDK 역량을 일부 반영해 평가한 점을 감안하면 **완전한 ○는 과합니다.**
→ 절충안: **○ → ◐**, 주석 "기본 앱은 단일 에이전트, **SDK로 멀티에이전트 구성 가능**".
이렇게 하면 사용자님의 지적과 보수적 평가 원칙을 동시에 만족합니다.

## 1.1.B Harness & Ops

### Q6. 다단계 권한(Permissions)이란?
**A. 행동의 위험도에 따라 자율성/승인 수위를 차등 적용하는 보안 체계입니다.**

핵심은 "허용/차단" 이분법이 아니라 **도구·명령 단위의 세분화된 정책**입니다. Claude Code를
예로 들면 두 축으로 동작합니다.
- **권한 모드:** `default`(쓰기·실행 시 확인) → `acceptEdits`(편집 자동 수락) →
  `plan`(읽기 전용 분석) → `bypassPermissions`(전체 자동, 위험).
- **규칙(rule):** 도구/명령 패턴별 `allow`·`ask`·`deny` (예: `Bash(git*)`는 allow,
  `Bash(rm*)`는 ask 또는 deny). MCP·웹페치 등도 개별 규칙화.

즉 "ls는 묻지 말고, 코드 수정은 diff 보여주고 승인받고, `rm -rf`·`npm publish`는 강하게
차단" 같은 **위험도 기반 단계화**가 다단계 권한입니다. Antigravity·Gemini 설명과 동일하며,
표의 "7권한"은 Claude Code가 이 정책 축을 특히 세밀하게 운용함을 가리킵니다.

### Q7. Cline의 Git 기반 checkpoint vs Claude Code rewind, 기술적 차이는?
**A. "파일만 되돌리는가 vs 파일+대화 컨텍스트를 함께 되돌리는가"가 핵심 격차입니다.**

| 항목 | Cline (Git 기반 checkpoint) | Claude Code (/rewind) |
|------|------|------|
| 복원 대상 | **파일 상태**(작업트리 스냅샷) | **파일 + 대화(컨텍스트) 상태** |
| 메커니즘 | 변경 전 스냅샷/임시 커밋 → 파일 되돌림 | 체크포인트에 코드·대화 함께 기록, 시점 선택 복원 |
| 의의 | 잘못된 *결과물*을 되돌림 | 잘못된 *추론 경로 직전*으로 AI 두뇌까지 되돌림 |

기술적 핵심: 에이전트가 엉뚱한 추론에 빠졌을 때, **파일만 복구하면 모델의 "오염된 기억"은
그대로** 남아 같은 실수를 반복합니다. Claude Code의 rewind는 **대화 상태까지 그 시점으로
되돌려** 다른 선택을 유도할 수 있습니다. 그래서 Claude ●, Cline ◐로 갈립니다.
(Antigravity·Gemini와 동일 결론. 다만 Gemini가 "Cline은 Git 없으면 작동 안 함"이라 단정한
부분은, Cline도 자체 스냅샷 경로가 있어 **Git 미사용 시에도 일부 동작**할 수 있으니 "Git 연동
시 견고"로 순화하는 게 정확합니다.)

### Q8. Git/Worktree 격리는 샌드박싱과 별개인가? 차이는?
**A. 네, 목적이 완전히 다른 별개 기능입니다.**

- **Git/Worktree 격리 = 버전관리(논리) 차원의 "코드 보호":** 에이전트를 별도 브랜치/
  워크트리에서 작업시켜 **메인 코드 오염을 방지**합니다. 실패하면 브랜치를 버리면 됩니다.
  보호 대상은 *내 코드/작업물*.
- **샌드박싱 = OS(물리) 차원의 "시스템 보호":** 실행 명령을 커널 수준(nsjail·Seatbelt·
  Landlock 등)에서 가둬 **파일시스템/네트워크 접근 자체를 차단**합니다. 보호 대상은
  *내 컴퓨터/계정 전체*. 악성 코드를 실행해도 격리되게 합니다.

요약: **Worktree는 "코드가 망가지는 것"을, 샌드박스는 "시스템이 털리는 것"을 막습니다.**
한쪽이 다른 쪽을 대체하지 못하므로 별도 지표가 맞습니다. (두 답변과 일치.)

## 1.1.C UX & Ecosystem

### Q9. Adaptive/개인화 vs 자기개선 학습 루프의 차이는?
**A. "사용자에 맞추는가(개인화) vs 스스로 더 잘하게 되는가(학습 루프)"의 차이입니다.**

- **Adaptive/개인화:** 대상은 **사용자**. 취향·환경·규약(코드 스타일, 린트, 선호 도구,
  추론 깊이 등)에 **행동을 맞추는** 것. 대체로 사용자가 명시(설정/`USER.md`)하면 반영.
- **자기개선 학습 루프:** 대상은 **에이전트 자신**. 작업을 풀어가며 얻은 성공 패턴을
  **재사용 가능한 스킬로 자동 승격**해 다음에 더 빨리/잘 푸는 폐루프(예: Hermes의
  Execute→Evaluate→Extract→Refine→Reuse).

핵심 구분선은 **"적응의 주체와 트리거"** 입니다. 개인화는 *사용자 신호→행동 조정*(주로 정적),
학습 루프는 *자기 경험→능력 진화*(동적·자율). 그래서 표에서도 둘을 별도 지표로 둡니다.
(Antigravity·Gemini와 일치.)

### Q10. Cline 코드엔 개인화 기능이 없어 보이는데 왜 높은 점수(●)인가?
**A. 사용자님 코드 분석이 맞습니다. ● → ◐(또는 ○) 하향을 권고합니다.**

이건 표의 가장 명백한 과대평가입니다. Cline의 "개인화"는 실제로는 **커스텀 인스트럭션
주입**(`.clinerules`/custom instructions를 시스템 프롬프트에 끼워 넣기)과 Plan/Act 토글,
프롬프트 자유도 정도입니다. 이는 **"도구 동작의 수동 튜닝(configurability)"** 이지,
사용자 행동·이력을 보고 **적응하는 알고리즘(adaptivity)** 이 아닙니다.

리뷰 과정에서 공식 문구의 "User Preferences"를 적응형 개인화로 후하게 해석한 것이 ●의
원인입니다. **"행동 적응 알고리즘 부재"** 라는 사용자님 판단을 반영해 **◐**(설정 자유도는
인정) 또는 엄격히는 **○**(진정한 adaptive 부재)로 내리는 게 정직합니다. 저는 **◐ + "정적
커스텀 인스트럭션 한정, 자동 적응 없음" 주석**을 권합니다. (Gemini도 ○/◐ 하향 동의.)

### Q11. 멀티 서피스/채널이란?
**A. "에이전트를 몇 개의 인터페이스(표면)로 접근할 수 있는가"입니다.**

- **서피스(surface):** CLI(터미널) / IDE 플러그인 / 데스크탑 앱 / 웹 등 **접근 창구**의 수.
  예) Antigravity는 CLI+IDE+Desktop+SDK의 4-surface → ●.
- **채널(channel):** Slack·Telegram·WhatsApp 등 **메신저 연동**으로 일을 시킬 수 있는가.
  예) OpenClaw·Hermes가 메시징 채널 강점 → ●.

즉 "어디서든(터미널이든 모바일 메신저든) 같은 에이전트에 접근 가능한가"의 지표이며,
이는 다음 항목인 **세션 핸드오프(기기·채널 간 무중단 이전)** 와 연결됩니다. 단, 서피스가
많아도 *상태가 끊기면* 핸드오프는 별도로 ○일 수 있어 두 지표를 구분합니다.
(두 답변과 일치.)

**메타 교훈:** 이번 질문들은 표의 구조적 약점 — *기능 유무*는 잘 담지만 *품질·체감*은
못 담는다 — 을 드러냈습니다. v0.6에서는 과대평가 셀을 하향하는 것과 더불어, 일부 핵심
셀에 **"문서 기준 / 체감 기준"을 구분하는 주석**을 다는 것을 제안합니다. 그래야 마케팅
claim과 실사용 reality의 간극이 표 자체에 정직하게 남습니다.
