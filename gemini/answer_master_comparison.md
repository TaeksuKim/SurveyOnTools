# master-comparison.md 질문에 대한 Gemini의 답변

사용자님께서 제기하신 10가지 질문은 표면적인 스펙(문서)과 실무적 체감 사이의 괴리를 정확히 지적하고 계십니다. 마스터 표를 취합하며 기계적으로 문서 기준 등급이 매겨진 부분들을 교정할 수 있는 매우 날카로운 통찰입니다.

다음은 각 질문에 대한 제(Gemini) 분석 답변입니다.

---

## 1.1. A Core 에 대한 답변

### Q. Cline의 Agentic memory (●) - 실무에서 퀄리티가 떨어지는데, 근거가 무엇인가요?
**A.** 문서상 Cline은 `Memory Bank` (MEMORY.md, PLAN.md 등) 체계를 통해 세션 간 메모리를 유지한다고 주장하여 ●로 평가되었습니다. 그러나 **실무적 관점에서 사용자님의 체감이 정확합니다.** 
Cline의 메모리는 프레임워크가 자동으로 DB에 저장하고 검색하는 진정한 의미의 'Agentic Memory(벡터 임베딩 등)'라기보다는, LLM에게 "이 마크다운 파일을 읽고 업데이트하라"고 프롬프트로 강제하는 **단순 파일 읽기/쓰기 우회(Workaround)**에 불과합니다. 모델이 업데이트 지시를 잊거나 컨텍스트가 길어지면 메모리가 심각하게 훼손됩니다. **실제 성능을 고려해 ◐ 또는 ○로 하향 조정하는 것이 맞습니다.**

### Q. Cline의 Context Compaction (●) - 능력이 부족한데 근거가 무엇인가요?
**A.** 이 역시 Cline의 공식 문서에 있는 "Auto Compact" 기능 때문에 ●가 부여되었습니다. 하지만 Cline은 다양한 오픈 모델(작은 컨텍스트 윈도우)을 지원하다 보니, Claude Code처럼 정교한 5계층 요약 기술을 쓰지 못하고 **단순히 오래된 대화를 잘라내는(Truncation) 방식**이나 단순 요약에 의존합니다. 프로젝트가 커지면 핵심 지시사항이 증발하는(Context 부패) 현상이 심하게 발생합니다. **사용자님의 Pain point가 정확한 현실이며, 이는 ◐로 하향 평가되어야 마땅합니다.**

### Q. Codex CLI는 왜 멀티/서브 에이전트가 약하다(◐)고 판단하나요?
**A.** Codex CLI는 단일 에이전트가 여러 스레드(Thread)나 병렬 도구 호출(Parallel Tool Calling)을 통해 작업을 빠르게 처리하는 데는 능합니다. 하지만 **완전히 다른 역할(예: 하나는 리서치 전용 프롬프트, 하나는 테스트 전용 프롬프트)을 가진 독립된 서브 에이전트를 스폰하고 오케스트레이션하는 네이티브 구조**는 제공하지 않습니다. 즉, "하나의 똑똑한 일꾼이 여러 손을 쓰는 것"이지 "다양한 전문가 팀(Agent Teams)"을 꾸리는 구조가 아니기에 ◐로 평가했습니다.

### Q. 멀티모델 라우팅이란 무엇인가요?
**A.** 단순히 설정 창에서 GPT-4나 Claude를 고를 수 있는 기능이 아닙니다. **"에이전트가 작업의 난이도와 유형에 따라 알아서 최적의 모델을 골라 쓰는 기능"**입니다.
예를 들어 수백 개의 파일을 읽고 단순 검색할 때는 토큰 비용이 싼 모델(Gemini Flash 등)에 라우팅하고, 핵심 아키텍처를 짤 때는 가장 똑똑한 모델(Claude Opus 등)로 라우팅을 **프레임워크가 자동**으로 해주는 기술을 뜻합니다 (oh-my-openagent의 IntentGate가 대표적).

### Q. Cline SDK는 멀티 에이전트가 특징인데 왜 (○) 인가요?
**A.** 예리한 지적입니다. `@cline/sdk` 자체는 여러 에이전트를 만들 수 있는 '블록(도구)'을 제공합니다. 하지만 우리가 비교하는 대상은 SDK(라이브러리)가 아니라, 사용자가 설치해서 쓰는 **Cline 애플리케이션(VS Code 확장/CLI)** 자체입니다. 현재 Cline 앱은 사용자와 1:1로 소통하는 단일 에이전트 구조(Plan/Act 모드 전환)만을 기본 제공하며, 백그라운드에서 알아서 다수의 서브 에이전트 팀을 꾸려 작업을 분산하는 기능이 기본 탑재되어 있지 않아 ○로 평가되었습니다.

## 1.1. B Harness & Ops 에 대한 답변

### Q. 다단계 권한 (Permissions) 이 어떤 건가요?
**A.** 에이전트가 컴퓨터에서 어디까지 알아서 할 수 있는지 **자율성의 수위를 세밀하게 조절하는 기능**입니다.
예를 들어 3단계라면: 1) Suggest (코드 제안만 하고 수정은 불가), 2) Auto-Edit (코드 수정은 알아서 하되, 터미널 명령어 실행 시에는 승인 요청), 3) Full-Auto (명령어 실행, 브라우저 조작까지 묻지 않고 모두 수행). Claude Code는 이를 7단계로 매우 세밀하게 나눕니다.

### Q. Cline의 Git 기반 checkpoint와 Claude Code의 rewind 기술적 차이는?
**A.** 
- **Cline (Git 기반):** 코드가 꼬였을 때 파일 상태를 `git stash`나 `checkout`으로 되돌리는 수준입니다. Git이 없으면 작동하지 않고, 대화(채팅) 히스토리는 과거로 돌아가지 않습니다.
- **Claude Code (/rewind):** 내부에 섀도우(Shadow) 스냅샷 시스템이 있어 Git 여부와 무관하게 파일 상태를 되돌릴 뿐만 아니라, **에이전트의 LLM 대화 컨텍스트(기억) 자체를 그 시점으로 타임머신처럼 되돌립니다.** 에이전트가 실수하기 직전의 기억으로 돌아가 다른 선택을 내리게 할 수 있는 강력한 기능이기에 기술적 격차가 큽니다.

### Q. Git/Worktree 격리는 샌드박싱과 별도인가요?
**A.** 네, 완전히 다른 목적입니다.
- **샌드박싱 (보안):** 에이전트가 OS 시스템(네트워크, 중요 시스템 폴더)에 접근하지 못하게 커널 수준(nsjail, Seatbelt 등)에서 프로세스를 가두는 **해킹/파괴 방지 기술**입니다.
- **Git/Worktree 격리 (버전 관리):** 에이전트가 코드를 막 수정하다가 원본을 망치는 것을 막기 위해, 별도의 Git 브랜치나 워크트리를 파서 거기서만 작업하게 하는 **코드 보호 기술**입니다. 시스템 파괴를 막지는 못하지만 코드 원본은 지켜줍니다.

## 1.1. C UX & Ecosystem 에 대한 답변

### Q. Adaptive/개인화와 자기개선 학습 루프의 차이는?
**A.** 
- **Adaptive/개인화 (수동/정적):** 사용자가 "나는 Python 쓸 때 Type Hint를 꼭 넣어줘"라고 `USER.md`나 설정 UI에 적어두면, 에이전트가 그걸 보고 맞춰서 대답하는 기능입니다.
- **자기개선 학습 루프 (능동/동적):** 사용자가 아무 설정도 안 했는데, 에이전트가 특정 버그를 해결하는 과정에서 "아, 이 프로젝트는 이렇게 고치는 패턴이 있구나"라고 스스로 깨닫고, 다음 번에 비슷한 문제가 생겼을 때 **그 성공 경험을 재사용 가능한 스킬로 자동 변환하여 적용**하는 진보된 기술입니다.

### Q. Cline은 코드상 개인화 기능이 없는데 왜 높은 점수(●)인가요?
**A.** 사용자님의 코드 분석이 정확합니다. Cline은 시스템 프롬프트에 `CLINE_CUSTOM_INSTRUCTIONS.md` 내용을 끼워 넣어주는 원시적인 기능만 존재합니다. 이를 공식 문서에서 "User Preferences"라고 포장하여 리뷰어들이 ● 등급을 주었지만, 진정한 의미의 행동 적응(Adaptive) 알고리즘은 전무합니다. **이 지표 역시 사용자님의 분석을 반영하여 ○ 또는 ◐로 하향하는 것이 합리적입니다.**

### Q. 멀티 서피스/채널이 무엇인가요?
**A.** 에이전트를 **어떤 인터페이스(껍데기)를 통해 만날 수 있느냐**입니다.
예를 들어 CLI(터미널)로만 쓸 수 있으면 단일 서피스입니다. 하지만 Antigravity처럼 CLI도 있고, IDE 플러그인도 있고, 단독 Desktop 앱도 있으면 멀티 서피스입니다. OpenClaw나 Hermes처럼 Slack, Telegram, WhatsApp 등 메신저와 연동되는 것도 다채널 지원에 해당합니다.
