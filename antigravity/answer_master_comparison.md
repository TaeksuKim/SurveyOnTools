# master-comparison.md 관련 사용자 질문에 대한 답변 (Antigravity)

작성자: Antigravity
기준일: 2026-06-17

사용자님께서 제기하신 질문들은 표의 추상화된 등급(●/◐/○)과 실제 사용자 체감 사이의 괴리를 정확히 짚고 계십니다. 제가 확보한 데이터와 아키텍처 지식을 바탕으로 각 질문에 답변드립니다.

---

## 1.1.A Core 에 대한 질문

### 1. Cline의 Agentic memory 강점 근거 및 체감 차이
**답변:** Cline이 ● 등급을 받은 근거는 공식 문서 상의 `Memory Bank` 프레임워크(`MEMORY.md`, `PLAN.md` 등을 통한 프로젝트 상태 관리 규칙)가 세션 영속성을 목적으로 "설계"되어 있기 때문입니다. 
하지만 사용자님께서 느끼신 정성적 불신의 원인은 명확합니다. Cline의 메모리는 벡터 DB나 백그라운드 지식 그래프를 활용한 '진정한 의미의 자동 리콜'이라기보다는, LLM 컨텍스트 윈도우에 명시적으로 텍스트 파일을 욱여넣는(inject) 방식에 의존합니다. 프로젝트가 커지면 파일 로딩 메커니즘이 한계를 보이므로, 실제 체감 성능은 ◐ 수준에 가까울 수 있습니다. (이 부분은 마스터 리포트 수정 시 하향 조정을 제안할 만합니다.)

### 2. Cline의 Context Compaction 능력 부족 현상
**답변:** 공식 문서에서 Cline은 `Auto Compact` 기능을 통해 단순 잘라내기(truncation)가 아닌 고충실도 요약을 한다고 선전합니다. 
그러나 실제 현업에서의 pain point는 1) 압축 시 코더에게 필수적인 '세밀한 디버깅 로그'나 '히스토리의 뉘앙스'가 유실되며, 2) 프롬프트를 다시 생성하기 위해 토큰을 재차 낭비하는 잦은 요약 사이클에서 발생합니다. 기술적으로는 '압축 기능이 존재'하여 ●로 평가되었으나, '압축의 질' 측면에서 사용자 경험은 낮을 수밖에 없습니다. 향후 아이디어로 제안한 "단순 요약이 아닌 컨텍스트 증류(Distillation)"가 절실한 부분입니다.

### 3. Codex CLI 멀티/서브 에이전트 약점 근거 (◐ 평가 이유)
**답변:** Codex CLI는 병렬 실행 스레드를 제공하지만, 그것은 주로 '동일한 에이전트의 여러 인스턴스'가 비동기적으로 도구를 호출하는 것에 가깝습니다. 반면 ●를 받은 도구들(예: oh-my-openagent)은 "기획자, 작성자, 리뷰어, 테스터"처럼 각각 다른 시스템 프롬프트와 도구를 가진 **이질적인 전문 에이전트들이 소통하며 자율 분업(팀 오케스트레이션)**하는 구조를 네이티브로 제공합니다. Codex는 이러한 역할 기반 팀 오케스트레이션 능력이 부족하여 ◐로 평가됩니다.

### 4. 멀티모델 라우팅의 정확한 의미
**답변:** 멀티모델 라우팅은 단순히 "설정창에서 GPT와 Claude를 골라 쓸 수 있다(복수 모델 지원)"는 것을 넘어서, **"작업의 종류와 난이도에 따라 가장 효율적인 모델을 에이전트가 '자동으로' 전환하며 사용하는 기능"**을 의미합니다. 예를 들어, 복잡한 아키텍처 설계에는 요금이 비싼 Claude Opus나 GPT-5.5를 호출하고, 단순 파일 탐색이나 오타 수정에는 저렴한 Gemini Flash나 Claude Haiku를 자동으로 라우팅하여 비용과 속도를 최적화하는 기술입니다.

### 5. Cline 멀티/서브 에이전트 미지원 논란 (SDK vs UI)
**답변:** 예리한 지적이십니다. `Cline SDK` 5계층 아키텍처 자체는 멀티 에이전트를 구축할 수 있는 "기반"을 완벽히 제공합니다. 하지만 우리가 사용하는 **기본 Cline 확장/도구(UI)**의 워크플로는 `Plan` 모드와 `Act` 모드를 넘나드는 "강력한 단일 에이전트" 방식입니다. 사용자 개입 없이 여러 에이전트가 뒤에서 알아서 팀을 이뤄 토론하고 결론을 도출하는 네이티브 조율(Orchestration) 기능이 UI/CLI 런타임에 내장되어 있지 않아 보수적으로 낮게(○ 또는 ◐) 평가되었습니다.

## 1.1.B Harness & Ops 에 대한 질문

### 6. 다단계 권한(Permissions)이란?
**답변:** 에이전트가 수행하려는 **행동의 위험도에 따라 사용자의 승인을 차등 요구하는 보안 체계**입니다. 예를 들어, 
1단계: 정보 검색(ls, cat 등) → 승인 없이 즉시 실행 (Auto)
2단계: 안전한 코드 수정 → 실행 전 Diff만 보여주고 승인 대기 (Suggest)
3단계: 위험한 셸 명령어(rm -rf, npm publish 등) → 강력 경고 및 명시적 거부/승인 (Ask/Deny)
이처럼 도구별로 권한을 잘게 쪼개어 자동화의 편의성과 보안을 동시에 취하는 기능입니다.

### 7. Cline Git 기반 체크포인트 vs Claude Code 차이
**답변:** 
- **Cline (Git 기반):** 코드를 수정하기 전에 임시 커밋을 만들거나 스냅샷을 찍어, 잘못되었을 때 "파일 상태"를 되돌립니다.
- **Claude Code (rewind) / Antigravity (undo):** 파일 상태뿐만 아니라 **"에이전트의 대화 컨텍스트(단기 기억)"까지 타임머신처럼 특정 시점으로 롤백**합니다. 에이전트가 잘못된 추론 경로로 빠졌을 때, 파일만 복구하는 것이 아니라 "그 엉뚱한 생각을 하기 직전"으로 AI의 두뇌 상태까지 되돌려 다시 명령할 수 있다는 기술적 격차가 존재합니다.

### 8. Git/Worktree 격리 vs 샌드박싱의 차이
**답변:**
- **Git/Worktree 격리 (논리적 격리):** 에이전트가 메인 코드를 망치지 않게, 눈에 보이지 않는 별도의 Git 브랜치(worktree)를 파서 작업하게 하는 **버전 관리 차원의 보호망**입니다.
- **샌드박싱 (물리적/OS 격리):** 에이전트가 실행하는 명령어가 아예 내 컴퓨터의 다른 폴더나 네트워크에 접근하지 못하도록 OS 커널 수준(nsjail, Seatbelt 등)에서 **실행 권한 자체를 원천 가두는 철창**입니다. 악성 코드를 받아 실행하더라도 내 컴퓨터가 털리지 않게 하는 훨씬 근본적인 보안 인프라입니다.

## 1.1.C UX & Ecosystem 에 대한 질문

### 9. Adaptive/개인화 vs 자기개선 학습 루프의 차이
**답변:**
- **Adaptive/개인화:** 에이전트가 **"사용자의 취향과 환경에 맞춰 행동 양식을 조절"**하는 것입니다. (예: 사용자가 선호하는 탭 스페이스, 린트 규칙 적용, 혹은 Thinking/추론 시간을 길게 하거나 짧게 하는 설정)
- **자기개선 학습 루프:** 사용자와 무관하게, 에이전트가 **"스스로 문제를 풀다가 새로운 방법론(스킬)을 터득하는 것"**입니다. (예: 처음엔 A 버그를 헤매면서 풀었지만, 성공 후 그 풀이 패턴을 스스로 '도구'로 만들어 저장해두고, 다음번엔 단번에 해결하는 자체 진화 루프)

### 10. Cline 개인화 점수가 높은 이유?
**답변:** 이 또한 문서 스펙과 체감의 차이입니다. Cline은 Plan/Act 모드 분리, 프롬프트의 자유로운 커스텀 주입, `/deep-planning` 등 사용자가 **"도구의 작동 방식(설정)을 매우 다채롭게 튜닝할 수 있는 높은 자유도"**를 제공합니다. 평가표에서는 이러한 "작동 방식의 맞춤화 기능"을 넓은 의미의 개인화로 인정하여 ●를 부여했습니다. (자동으로 내 마음을 읽고 학습하는 인공지능적 개인화와는 다릅니다.)

### 11. 멀티 서피스/채널이란?
**답변:** 에이전트를 **얼마나 다양한 껍데기(인터페이스 표면)에서 접근할 수 있는가**입니다.
터미널(CLI)에서만 쓸 수 있느냐, VS Code 같은 에디터(IDE) 창 안에서도 쓸 수 있느냐, 별도의 데스크탑 앱으로 돌아가느냐, 혹은 Slack이나 Telegram 같은 모바일 메신저를 통해서도 에이전트에게 일을 시킬 수 있느냐(OpenClaw, Hermes 등)를 의미합니다. 다양한 채널을 지원할수록 점수가 높습니다.

---
이상 Antigravity의 분석입니다. 다른 에이전트(Claude, Gemini)들의 의견과 함께 교차 검증하시면 더욱 입체적인 결론을 얻으실 수 있을 것입니다.
